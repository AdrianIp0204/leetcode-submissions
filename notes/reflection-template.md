# Reflection Template

Use this after every accepted solution, even if the problem was easy.

```md
## Pattern

One short pattern label, such as hash map, stack, two pointers, prefix/suffix, linked list pointer rewiring, binary search, DFS/BFS, dynamic programming, or brute force baseline.

## Key Idea

Two to five sentences. Explain the invariant or trick in your own words. If the accepted solution is brute force or brittle, say that instead of pretending it is the best approach.

## Mistake / Edge Case

What almost broke the solution? What would you check first next time?

## Complexity

- Time: O(...)
- Space: O(...)

## What Adrian Should Remember

One concrete lesson for next time.
```

## Quality Bar

- Do not paste the LeetCode problem statement.
- Do not write a generic explanation that could fit any problem.
- Mention if the code is accepted but not ideal.
- Keep it readable enough that future Adrian can review it quickly.
- If Morrow writes the first draft, Adrian should mark anything that does not match his real understanding.
